# Claude Feedback Protocol

## Purpose
This document defines the protocol for managing inline feedback comments to prevent duplicate work by Claude and agents.

## Comment Syntax

### Active Feedback (Unresolved)
```markdown
[@Claude do something here]
[@agent-name do something specific]
```

### Resolved Feedback
Use STRIKETHROUGH to mark resolved feedback:
```markdown
~~[@Claude do something here]~~
```

## Protocol Rules

### For Claude/Agents:
1. **ALWAYS search for active feedback** before starting new work:
   ```bash
   grep -r "\[@Claude" --include="*.md" | grep -v "~~\[@"
   ```

2. **IGNORE struck-through feedback** - it's already resolved:
   - Pattern: `~~[@Claude ...]~~`
   - These represent completed work and should not be re-executed

3. **Mark feedback as resolved** after completing the work:
   - Replace `[@Claude ...]` with `~~[@Claude ...]~~`
   - Keep the comment for history/context
   - Document what was done in the commit message

### For Humans:
1. **Write new feedback** using the `[@Claude ...]` format
2. **Let Claude resolve** feedback by converting to strikethrough
3. **Search for unresolved** feedback anytime:
   ```bash
   # Find all active feedback across the project
   grep -rn "\[@Claude" --include="*.md" | grep -v "~~"
   ```

## Examples

### Before Resolution
```markdown
The metrics are solid.

[@Claude update the KPIs/metrics in the Brief to match this feedback and make measurable trust a priority for everyone]

Next steps...
```

### After Resolution
```markdown
The metrics are solid.

~~[@Claude update the KPIs/metrics in the Brief to match this feedback and make measurable trust a priority for everyone]~~

Next steps...
```

## Automated Checking

Claude will run this check at the start of sessions:

```bash
# Check for unresolved feedback
grep -rn "\[@Claude" docs/ Instructions/ --include="*.md" | grep -v "~~\[@"
```

This ensures no feedback comments are missed or duplicated.

## Alternative: Clean Deletion

If you prefer to keep files clean, you can also just DELETE resolved comments entirely. The work will be captured in git history. This is simpler but loses inline context.

Choose whichever style you prefer - Claude will adapt to your preference.
