S1 Lexical standard case — P227657 · ATF + matched translations · token_readings pending (placeholder data shown)

Edge Cases Visible in This Tablet

Reconstructed tokens [bracket]
Line 5 and 15: `[tu₇]` and `[x x]` are restored readings. The current token design renders these the same width as intact tokens — no visual affordance communicates the epistemological difference. Scholars need to know restoration confidence.
Compound damage (#!?) has no triage
Line 9: `SI#!?` means damaged + editorially corrected + uncertain. Three distinct states collapsed into a single token. The current flag area only shows "DMG" and "UNC" but not the combination hierarchy or scholarly attribution.
Determinatives are not first-class tokens
Line 12: `{gi}` (reed determinative) is a semantic classifier, not a spoken word. Displaying it identically to lexical tokens misrepresents the cuneiform grammar and inflates word counts.
No translation language toggle
P227657 only has English. But multi-language tablets (P135710: en+de, P131589: en+fr) need a language selector somewhere in the header. The current design has no affordance for this.
Competing lemma indicator (•) has no interaction
Line 9 ASZ token has multiple viable lemmatizations. The amber dot is present but clicking it does nothing in the design. Scholars routinely need to see and adjudicate competing readings.
Long translations overflow the bar
Lines 8–9: translations like "soup made with the flour of the ... pulse" are 7+ words. The single-line translation bar clips or wraps awkwardly. No design spec for multiline translation display.

S2 Damaged Sumerian — P247526 · fully broken lines, damage markers, sub-line notation (11.a / 11.b), unmatched translation

Edge Cases Visible in This Tablet

Fully broken lines have no tokens
Lines 1–2 are `[...]` — no recognisable content. The token ribbon renders a single `[...]` placeholder token. But the Translation bar shows "... ..." which is technically a translation. Should broken lines even have a translation bar? The status dot system (●/○/⋯) breaks down here.
Sub-lines (11.a, 11.b) break the column model
Administrative texts routinely use sub-line notation. The current design has no indent hierarchy — sub-lines render at the same weight as parent lines, destroying the ledger-like structure scholars use to understand accounting records.
Unmatched translations: silent gap
Line 13 has no matched translation. The empty translation bar is the only signal, but it gives no reason (pipeline gap? genuinely untranslated? broken text?). Scholars need to distinguish these states.
Multi-word compounds as atomic units
Sumerian constructs like `udu niga saga` (3 tokens) translate as one semantic unit "good quality fattened sheep." The individual glosses (sheep / fattened / fine) don't reconstruct that meaning. Scholars need compound-level annotation in addition to token-level.
No surface_type = no surface tabs
P247526 has null surface_type for all lines. The Obverse/Reverse tabs still render — but clicking Reverse would show nothing. The UI should handle missing surface data gracefully, hiding the tab or showing "unseparated" state.
Damage affects translation confidence
When a key token like `udu#` is damaged, the translation "fattened sheep" is actually inferred. There is no visual link between token damage level and translation confidence. Scholars need this connection visible.

S3 Complex ATF — P229672 · fragment join · composite Q000057 · mid-sentence %a language switch · structural $-lines · prime notation · seal impression

Edge Cases Visible in This Tablet

Language switch: token alignment breaks
Bilingual lines have a Sumerian half and an Akkadian half. They are NOT parallel — 5 Sumerian tokens ≠ 3 Akkadian tokens. The current fixed-width token grid can't convey this asymmetry. The `%a` boundary marker is present but has no semantic weight in the design — it looks like another token.
Structural tokens are rendered as words
The `~` (equivalence), `=`, and `(blank)` in Diri entries are ATF structural operators, not lexical tokens. Rendering them in the same token grid as `diri` makes it look like they are words to be lemmatized — they are not. No design spec handles non-lexical token types.
Determinative {d} (divine) inside a token
Line 9: `{d}en-lil₂` — the `{d}` is a divine name determinative embedded in a god's name. It must render as superscript. The token model treats it as part of the reading string, losing the typographic convention that scholars rely on.
"Rest broken" creates orphaned columns
When `$ rest broken` appears, the column terminates. The remaining visual space in the 3-column grid is empty — currently indistinguishable from "data not yet loaded." Scholars need a visual termination marker that communicates physical damage, not system error.
Prime notation (8') has no explanation
Lines with `'` suffix are interpolated or damaged lines from a parallel source. The prime number is displayed in the line number gutter but there is no tooltip or indicator explaining what "primed" means — critical for students and non-specialists.
Fragment join in designation is not parsed
"K 03254 + K 03779" signals a physical join of two tablet fragments. The Join badge is present, but clicking it does nothing. Scholars need to see: which museum holds each fragment, when the join was identified, and whether the join is certain or proposed.

Improvement Opportunities

Drawn from domain research, ORACC workflows, and scholar persona analysis

Critical Gap

Damage-to-Translation Confidence Chain

When tokens are damaged (#), the translation is inferential. The current design shows a green ● for matched translations even when the source line is substantially broken. Scholars need a visual confidence score on the translation bar that degrades with token damage — e.g. amber ● for partially damaged, red ● for mostly reconstructed.

Critical Gap

Translation Language Selector

Multiple tablets have EN+DE, EN+FR, or EN+TS (normalized) translation layers. The current design has no header affordance for toggling translation language. This must be in the header — alongside the surface tabs — before launch, since P135710, P131589, P258950 all require it.

Scholar Workflow

Competing Lemmatization Adjudication

ORACC corpora regularly have 2–3 competing lemmatizations for ambiguous signs. The amber dot is present in the design but clicking it should expand an inline panel showing: each candidate citation form + guide word, the scholar/tool that proposed it, the source publication, and a "mark consensus" affordance. Without this, the design is read-only for the most contested data.

Scholar Workflow

Interlinear vs. Ribbon Mode Toggle

The current ribbon design is excellent for reading. But for line-by-line philological work (comparing token-level readings against a published edition), scholars need a stacked interlinear mode where each layer (ATF / NORM / LEMMA / GLOSS) appears on its own row below the tokens. A view toggle in the column header would serve both audiences.

Data Layer

Provenance Tooltip on Every Layer

The Glintstone schema records who/what created each annotation (scholar_id, annotation_run, publication_ref). This is unused in the UI. Hovering any token row (reading, norm, lemma, gloss) should show: "Added by [Name], [Year], via [method]." This is the core trust-infrastructure differentiator from CDLI and ePSD2.

Data Layer

Sub-Line Hierarchy in the Row Design

Administrative texts (Ur III, OB) use sub-line notation (3, 3.a, 3.b) to encode ledger structure. The current flat row design destroys this. Sub-lines should render indented with a connecting bracket or vertical rule linking them to their parent — similar to how code editors handle code folding.

UX Gap

Structural Lines Need a Distinct Render

$-lines (§ beginning broken, § single ruling, § seal impression) currently appear as visual dividers or notes between rows. But they have no interaction and no information hierarchy. A collapsed pill showing "§ 3 editorial notes" with expand-on-click would reduce visual noise while preserving access to the ATF structure.

UX Gap

Determinative Superscript Rendering

Cuneiform determinatives ({d} divine, {ki} place, {m} personal name, {f} female, {gi} reed) are semantic classifiers that render as superscript in all published editions. Rendering them as full-size tokens inflates word count and violates typographic convention. They should appear as small superscript prefixes attached to the following token, not as independent columns.

UX Gap

Bilingual Zone Visual Grammar

The current language-zone bars (green = Sumerian, blue = Akkadian) are a good start, but the relationship between the two zones in a bilingual line is ambiguous. Is the Akkadian a translation of the Sumerian, or an equivalent reading of the same text? A directional arrow or "=" label between zones, plus zone labels (SUM / AKK), would clarify this for scholars working across traditions.

Scholar Workflow

Keyboard Navigation Between Tokens

Philologists work line-by-line through a text, moving from token to token. The current design is pointer-only. Arrow key navigation (left/right through tokens, up/down between lines) with auto-scroll would dramatically speed up reading sessions. This is how Logos Bible Software and BibleWorks handle interlinear navigation.

Data Layer

Attestation Count on Glosses

ePSD2 shows attestation frequency for every lemma (e.g. "ninda[bread]N — 33,019 attestations"). This frequency data exists in the Glintstone glossary_entries table (icount). Adding a subtle count below the gloss row ("×33k") gives scholars immediate frequency context — essential for evaluating whether a reading is standard or rare.

Scholar Workflow

Composite Text Cross-Reference

P229672 is an exemplar of composite Q000057. When viewing this tablet, scholars typically want to compare it against other exemplars of the same composition. The composite badge should open a panel showing: all known exemplars of Q000057, their damage state, period, museum, and direct line-by-line comparison links. This is the core use case for comparative textual criticism.
