# PRD: Agentic Summarization & Translation Suggestion System

## North Stars

Two user jobs. Everything in this PRD exists to serve them.

| Job | User question | Feature |
|---|---|---|
| Triage | "Which artifacts should I focus on?" | Artifact summaries in sidebar + search snippets |
| Comprehension | "What does this line say?" | Line-level translation suggestions in ATF viewer |

---

## What Is Already Built

- **`summarize_artifact` API** — `GET /api/v2/artifacts/{p}/summary?focus=…` — implemented in `core/agent/fact_assembly.py` + `core/agent/synthesis.py`, with citation validation, lazy-persist via `agent_outputs`, debug UI at `_debug/agentic.html`.
- **`interpret_token` API** — `GET /api/v2/artifacts/{p}/tokens/{id}/interpret` — implemented; returns token chain + 1–3 hypotheses. Line suggestions sit *above* this; they share the same infrastructure.
- **Schema** — `agent_outputs`, `agent_interactions`, `interaction_feedback`, `entity_embeddings` in migration 025. `agent_outputs.model` tracks model + version. **Known gap: `model` is not in the unique index — see §Data Model Extensions.**
- **Synthesis loop** — grounded citation enforcement, retry-on-failure, `best_guess_flag`, `prompt_version` versioning. Retry logic has a known flaw for JSON-output prompts — see §Prompt Architecture.
- **Prompts** — `prompts/synthesis.v1.md`, `prompts/interpret-token.v1.md`.

Phase 1 wires the existing API to the UI and fixes known gaps. Phase 2 adds `suggest_line_translation` as a net-new tool.

## Phase 1: Artifact Summaries

### 1.1 Sidebar Refactor

**Current state:** Knowledge sidebar has a Dictionary tab with filters and a list.

**Target state:** Single **Summary** tab. Dictionary tab removed from DOM (not deleted — future tabs are additive). No other tabs until user research justifies them.

**Summary tab layout (top to bottom):**

1. **Pipeline completeness bar** — 0–5 visual indicator from `pipeline_completeness` view. Stages: ATF / Tokens / Lemmatization / Translation / Entities.
2. **Language + dialect badge** — from `surfaces.lang` + dialect fact. Shows "Sumerian," "Akkadian (Old Babylonian)," "Mixed: Sumerian + Akkadian," etc. For unsupported languages, badge reads "AI suggestions not available for [language]."
3. **Hypothesis notice** — when `best_guess_flag = true`, a visually distinct callout *before* the summary text: "This tablet has sparse data. The following is inferred from similar tablets." Never inline.
4. **Summary text** — 2–5 sentences. Skeleton loader on first generation (< 3 s); cached hit < 200 ms.
5. **AI attribution badge** — small muted "AI" chip with tooltip: "Generated by AI from available scholarly data. Not peer-reviewed." Never styled as a citation.
6. **Thumbs feedback** (Level 1) — thumbs up / down on the summary block.
7. **Selected line context** (Phase 2, same tab) — when a line is selected in the ATF viewer, the suggestion and token chain appear below the summary. Same tab, no navigation needed.

**State management:** Sidebar opens collapsed by default. On open, fetch if not cached in DOM. DOM cache per page session.

### 1.2 Search Result Snippets (Option C)

- Search results show catalog metadata by default — no change.
- If a non-superseded `agent_outputs` row exists for a P-number, surface the first 1–2 sentences as a snippet.
- No on-demand generation from search. Summaries accumulate from detail-page visits.
- AI badge on every snippet.

### 1.3 Batch Summarization

```bash
python -m core.agent.batch summarize \
  --filter period="Ur III" \
  --filter min_completeness=3 \
  --limit 500 \
  --focus general \
  --concurrency 4
```

Or via `POST /api/v2/internal/agent/batch-summarize` with a filter JSON body.

**Behavior:** Idempotent. Skips artifacts with valid cached `agent_outputs`. Writes progress to stdout and a `batch_log` JSONL. Trigger (cron, manual, admin UI) is TBD — the capability ships; scheduling is a later decision.

### 1.4 Fact Assembly Extensions (Phase 1 fixes to `assemble_artifact_facts`)

These fix gaps in the existing assembler before wiring it to the UI. All changes are to `core/agent/fact_assembly.py`.

**a. Genre normalization — join through `genre_canon`, not `artifacts.genre` directly.**
Raw genre values in CDLI are inconsistently cased and abbreviated. `genre_canon` normalizes them. Also pull `subgenre` and `supergenre` where non-null — they are high-signal for translation register.

**b. Language + dialect fact.**
Query `surfaces.lang` (primary) with `artifacts.language` as fallback. Join to `glossary_entries.dialect` where populated. Render as: `primary language: Sumerian (sux)` or `primary language: Akkadian (akk), dialect: Old Babylonian`. Add period gloss for non-specialist model benefit: `period: Ur III (ca. 2112–2004 BCE, southern Mesopotamia)`.

**c. Mixed-language flag.**
Detect multiple `%lang` markers across `text_lines.atf_language` for the artifact. When `%akk` and `%sux` both appear, add: `mixed-language text: Sumerian and Akkadian sections present`. Extend to handle 3+ languages (Hittite ritual texts can have three).

**d. Composite sibling data.**
When a tablet is part of a composite (Q-number), pull the best-attested exemplar's completeness score and top 5 lemmas. Add as facts: `composite sibling [Q-number] has lemmatization and translation — genre signal available`. This closes the biggest gap for sparse tablets that are part of well-attested composites.

**e. Fix the best-guess gate.**
Current gate: `completeness ≤ 2 AND ≥3 similar tablets above cosine 0.72`. Broken: sparse tablets have no lemma_bag embedding, so cosine never reaches 0.72. Fix: fall back to designation-text embedding when lemma_bag is absent; lower threshold to 0.65 for the sparse path.

**f. Source attribution in rendered facts.**
Append `(CDLI catalog)`, `(ORACC)`, or `(user-correction)` to each rendered fact line so scholars can verify `[n]` citations without looking up annotation_run IDs.

**g. Competing lemmatizations.**
When multiple `annotation_runs` carry different `lemma_id` values for the same token, add a fact: `competing reading for token X: scholars disagree — analysis A (run #44) vs. analysis B (run #51)`. Lets the model hedge explicitly.

**h. Trust signal.**
When `annotation_runs.trust < 0.7`, append `(uncertain)` to the rendered fact. Grounds model hedging in data quality, not guesswork.

### 1.5 Synthesis Prompt v2 (`synthesis.v2.md`)

`synthesis.v1` has no dialect, mixed-language, or trust instructions. `synthesis.v2` adds:

- **Dialect rule:** "If dialect is Old Babylonian, note it differs from Standard Babylonian. Do not use dialect-specific vocabulary inference unless a FACT supports it."
- **Mixed-language rule:** "If the mixed-language fact is present, acknowledge code-switching. Do not assign a uniform genre register."
- **Register guidance:** Royal inscriptions, administrative tablets, and literary texts require different prose registers. Tune to the genre fact.
- **Competing readings rule:** "If a competing-reading fact is present, hedge with 'scholars disagree on' rather than stating the reading as settled."
- **Uncertainty propagation:** "If a fact carries `(uncertain)`, reflect that uncertainty in the claim — use 'possibly' or 'tentatively attributed to'."
- **Focus output templates:** `translation_status` focus must enumerate missing pipeline stages in order of completion difficulty, not just note completeness score.

Prompt versioning: `synthesis.v2.md` in `prompts/`. Old `synthesis.v1` outputs remain valid until a new annotation_run triggers regeneration — no forced sweep.

### 1.6 Language Scope

**In-scope v1:** `sux`, `akk`, `akk-x-stdbab`, `akk-x-oldbab`, `akk-x-neoass`, `akk-x-neobab`, `akk-x-ltebab`, `akk-x-middleass`, `akk-x-midbab`, and Sumerian+Akkadian mixed.

**Out of scope v1:** `elx`, `hit`, `hur`, Ugaritic, others. UI badge: "AI suggestions not available for [language name]." Documented in `gs-expert-assyriology/languages.md`.

**API response includes `language_supported: bool`** so the web layer gates without hardcoding language codes.

## Phase 2: Line Translation Suggestions

### 2.1 Tool: `suggest_line_translation`

Sits *above* `interpret_token` (token-level) — operates on a whole line, proposes coherent English translations, and exposes the token chain inline.

**REST:**
```
GET /api/v2/artifacts/{p_number}/lines/{line_id}/suggest
    ?variant=a|b
```

**MCP:**
```python
suggest_line_translation(
    p_number: str,
    line_id: int,
    variant: Literal["a", "b"] | None = None,
) -> ToolResponse[LineSuggestionPayload]
```

Variant assignment: random per session if omitted; logged in `agent_interactions.request.variant`. A/B test is embedded in the tool itself — no separate experiment infrastructure needed.

### 2.2 Fact Assembly: `assemble_line_facts`

New function in `core/agent/fact_assembly.py`. Critical design decisions from analysis:

**Neighbor context = ±2 lines, not ±3 tokens within the line.**
The existing `interpret_token` assembler uses `ABS(t.position - target_position) <= 3` — this gives tokens on the *same line*. The useful context for line translation is *neighboring lines*. A standalone line means nothing; surrounded by a letter formula it's unambiguous. Fix: query `text_lines` for lines at `line_number ± 2`, including their ATF and any available lemmas.

**Genre prior from a materialized view.**
The PRD originally described genre prior as pulled from `entity_embeddings` — that entity type doesn't exist yet. Create `genre_period_lemma_prior` as a materialized view before Phase 2 ships:

```sql
CREATE MATERIALIZED VIEW genre_period_lemma_prior AS
SELECT
    a.genre_normalized,
    a.period_normalized,
    ll.citation_form,
    ll.guide_word,
    COUNT(*) AS attestation_count,
    RANK() OVER (
        PARTITION BY a.genre_normalized, a.period_normalized
        ORDER BY COUNT(*) DESC
    ) AS rank
FROM artifacts a
JOIN surfaces s ON s.p_number = a.p_number
JOIN text_lines tl ON tl.surface_id = s.id
JOIN tokens t ON t.line_id = tl.id
JOIN lemmatizations lz ON lz.token_id = t.id
JOIN lexical_lemmas ll ON ll.id = lz.lemma_id
WHERE a.genre_normalized IS NOT NULL
  AND a.period_normalized IS NOT NULL
GROUP BY a.genre_normalized, a.period_normalized, ll.citation_form, ll.guide_word
WITH DATA;

CREATE INDEX ON genre_period_lemma_prior(genre_normalized, period_normalized, rank);
```

Refresh nightly or after major ingestion runs. Pull top 20 by rank for the target genre×period.

**Determinative extraction.**
ATF determinatives (`{d}`, `{ki}`, `{m}`, `{f}`, `{uruda}`, etc.) appear in the token sequence. Extract them and surface as explicit facts: `token 3 carries divine determinative {d} — likely a divine name or theophoric element`. This is the single highest-confidence signal for named entity type and currently invisible to the model.

**Sign candidate lookup fix.**
Current code splits the raw form on `[-.]` and looks up each fragment independently. For syllabic Akkadian (`ṭup-pi-im` → `["ṭup", "pi", "im"]`), this produces meaningless individual syllable lookups. Fix: branch on language mode. Sumerian logograms split by `-`; Akkadian syllabic spellings → normalize to citation form first, look up the whole form. Sumerian compound signs (`dingir-ra`) split correctly.

**`missing_layers` is server-computed, not model-generated.**
Computed from the fact bundle before the LLM call: which pipeline stages are absent for the tokens in this line. Injected into the payload post-synthesis. Not a field the model produces. Removes ambiguity and one source of hallucination.

**Language shift markers.**
`%akk` and `%sux` appearing in `text_lines.atf_language` mid-line indicate code-switching. These are metadata tokens in the token sequence. The assembler must detect them (they appear in the raw ATF between word tokens) and add an explicit fact: `language shift at token position N: Sumerian → Akkadian`. The model cannot reason about code-switching without this signal.

**Full fact list for `assemble_line_facts`:**

| Fact | Source | Variant |
|---|---|---|
| Target line ATF | `text_lines.atf_transliteration` | Both |
| Language + dialect | `surfaces.lang`, `glossary_entries.dialect` | Both |
| Language shift markers | detected from raw ATF | Both |
| Surface label (obv/rev/edge) | `surfaces.surface_type` | Both |
| Token chain (form → lemma → guide_word → Akkadian morphology) | `tokens` + `lemmatizations` + morphology tables | Both |
| Determinative extraction | parsed from raw ATF | Both |
| Neighbor lines ±2 with lemmas | `text_lines` | Both |
| Artifact catalog (period with date gloss, provenience, genre, subgenre) | `artifacts` + `genre_canon` | Both |
| Genre×period prior (top 20 lemmas) | `genre_period_lemma_prior` view | Both |
| Named entities on this line | `entity_mentions` → `named_entities` | Both |
| Existing partial translation for this line | `translations` where `line_id` matches | Both |
| Competing lemmatizations | multiple `annotation_runs` per token | Both |
| Full surface ATF (truncated) | all `text_lines` for the surface | B only |

**Variant B surface truncation:** Keep first 800 chars + last 400 chars, `[... N lines omitted ...]` in the middle. Royal inscriptions and literary texts carry their most identifying content at the *end* (colophons, closing formulae) — end-truncation loses exactly that.

**Variant B genre gating:** Only use variant B for `literary`, `letter`, `royal inscription`, and `lexical` genres — the genres where surface structure is formulaic and the extra context adds meaningful signal. Default to variant A for `administrative` (the majority of the corpus). Administrative tablets are lists; the genre prior captures the same information at lower cost.

### 2.3 `LineSuggestionPayload` Schema

```python
class TokenChainStep(BaseModel):
    token_id: int
    raw_form: str
    is_determinative: bool              # True for {d}, {ki}, etc.
    lemma: str | None                   # citation_form if lemmatized
    guide_word: str | None
    pos: str | None
    # Akkadian morphology (when populated in lemmatizations table)
    case: str | None                    # nominative, genitive, accusative, etc.
    number: str | None                  # singular, plural, dual
    gender: str | None                  # masculine, feminine
    translation_fragment: str | None    # NULLABLE — null when confidence_band is "unknown"
    fact_refs: list[int]
    confidence_band: Literal["known", "inferred", "unknown"]

class LineSuggestion(BaseModel):
    rank: int
    translation: str
    confidence_band: Literal["low", "medium", "high"]
    evidence_chain: list[str]           # 1–3 fragments; must cite [n] inline
    fact_refs: list[int]
    caveat: str | None

class LineSuggestionPayload(BaseModel):
    line_id: int
    atf: str
    language: str
    dialect: str | None
    is_mixed_language: bool
    language_shift_position: int | None  # token index where shift occurs, if any
    language_supported: bool
    token_chain: list[TokenChainStep]
    suggestions: list[LineSuggestion]   # 1–3, ranked; derived from token_chain
    missing_layers: list[str]           # server-computed, not model-produced
    context_variant: Literal["a", "b"]
    model: str                          # which model generated this
    prompt_version: str
```

### 2.4 Prompt: `suggest-line.v1`

New file: `prompts/suggest-line.v1.md`.

**Critical instruction: token chain first, suggestions derived from it.**
"Generate `token_chain` first. Your `suggestions` must only use readings marked `known` or `inferred` in `token_chain` — never introduce a reading in `suggestions` that doesn't appear in `token_chain`." Without this, smaller models drift between the two sections and produce internally inconsistent output.

**`translation_fragment` is nullable.**
"If `confidence_band` is `unknown`, set `translation_fragment` to null. Do not guess at a fragment for tokens with no evidence."

**Target line disambiguation (critical for smaller models).**
The user message separates the target line from surface context explicitly:

```
TARGET LINE (translate this):
[2] obv. 4: i-na ṭup-pi-im ša-a-ti

SURFACE CONTEXT (use for context only — do not translate these lines):
[5] obv. 1: ...
```

Without this separation, Gemini Flash and Llama models will occasionally translate a neighboring line instead.

**Mixed-language instruction.**
"If a language shift fact is present, note the shift in the relevant suggestion caveat. Do not apply Sumerian grammar rules to Akkadian tokens or vice versa."

**Same guard rails as `interpret-token.v1`:** "resembles / consistent with / may correspond to" — never "this line means X." Same `[n]` citation contract. Same reject-retry logic (see §Prompt Architecture for JSON retry fix).

### 2.5 UI Integration in ATF Viewer

**Primary trigger — lazy load on page mount:**
On ATF viewer mount, identify untranslated lines in the initial viewport. Fire background requests, up to 5 at a time, staggered 50 ms apart. Cap at **5 lines per initial viewport** (not 10) — this cuts first-load API cost in half with negligible UX impact. As suggestions arrive, update each line's DOM.

Scroll listener: enqueue visible untranslated lines not yet fetched; batch in groups of 5.

**Fallback trigger — on-demand click:**
Lines not yet fetched show a ghost affordance on hover. Click triggers an explicit fetch with a loading state.

**ATF viewer visual:**
```
│ obv. 4  |  i-na ṭup-pi-im ša-a-ti    [Akkadian OB]
│          ↳ [AI] in/on that tablet  (medium)
│              expand ↕ → shows token chain + all suggestions + missing layers
```

- `[AI]` badge in muted style. Top suggestion inline. Expand reveals full chain.
- Expanded: token chain shows each token's lemma or `unknown`, determinative flags, missing_layers plain-language summary.
- For language_supported = false: "AI suggestions not available for [language]."

**Sidebar integration:**
Line click in ATF viewer → sidebar Summary tab updates below the artifact summary to show the selected line's suggestion + token chain. No tab navigation needed.

### 2.6 A/B Test

- Assignment: random per session, stored in JS session variable, passed as query param, logged in `agent_interactions.request`.
- Hypothesis: B wins for literary/letter/royal inscription; A adequate for administrative.
- Measurement: Voyage cosine against withheld reference translations, segmented by genre.
- Outcome drives per-genre defaults post-eval.

## Prompt Architecture

### Universal Three-Part Structure

Every prompt, every scenario, every provider follows the same anatomy:

```
┌─────────────────────────────────────────────┐
│  SYSTEM PROMPT (~600–800 tokens, static)    │
│  Role + grounding rules + output contract   │
│  Cached where supported                     │
├─────────────────────────────────────────────┤
│  FACTS BLOCK (dynamic, 300–2000 tokens)     │
│  Numbered [n] claims assembled from DB      │
│  Identical format across all providers      │
├─────────────────────────────────────────────┤
│  INTENT LINE (~10–30 tokens)                │
│  What to do + modifiers ([BEST_GUESS] etc.) │
└─────────────────────────────────────────────┘
```

The `[n]` numbering is the connective tissue across all three scenarios. Facts are numbered; the model cites them; the validator enforces round-trip integrity. This contract is identical across providers.

### Provider-Specific JSON Mode

The JSON output contract (for `suggest_line_translation`) adapts per provider:

| Provider | Mechanism | Notes |
|---|---|---|
| Anthropic | Prompt instruction | Already working in interpret-token.v1 |
| Gemini Flash | `response_mime_type: "application/json"` + schema | Most reliable; use native JSON mode |
| Groq / Together / OpenAI-compatible | `response_format: {"type": "json_object"}` | Reliable; one param |
| Llama via Ollama | Prompt instruction + one-shot example in system | Add ~150-token example to system prompt |

The `OpenAICompatClient` adapter passes `response_format` when a prompt signals JSON output. One adapter covers Groq, Together, Fireworks, and Gemini (via their OpenAI-compatible endpoint).

### Retry Logic Fix for JSON Prompts

**Current behavior (broken for JSON):** On citation failure, `synthesis.py` appends the full failed output + complaint to the user message. For prose this is fine; for JSON, the retry message becomes structurally confusing — the failed JSON blob looks like a new instruction to smaller models.

**Fix:** For JSON-output prompts, the retry sends only the complaint and the specific failing field, not the full failed JSON:

```
Previous attempt failed validation:
  Problem: fact_refs contains [15] which is not in the FACTS list.
  Affected field: suggestions[0].fact_refs

Re-generate the JSON. All other fields were correct — only fix the flagged field.
```

This is a targeted patch to `core/agent/synthesis.py`, gated by a `json_output: bool` flag on the synthesis call.

### `[n]` Citation Compliance Across Providers

Claude's first-pass citation compliance rate is ~97%. Gemini Flash and Llama 3.3 70B will be lower — smaller models drop markers on short facts or collapse multiple citations. This will be measured directly in the eval (tertiary metric: grounding compliance rate). If compliance falls below 85% on first attempt for any provider, the system prompt for that provider gets an additional example section showing correct vs. incorrect marker usage.

### Embedding Invalidation Fix

When a new `annotation_runs` row is created for an artifact, the `entity_embeddings` lemma_bag for that artifact is stale — but it isn't invalidated. The similar-tablet retrieval uses the old embedding until the nightly cron runs. Fix: when the `agent_outputs` sweep marks a summary as superseded, also queue the artifact's embeddings for priority refresh in the backfill cron. The cron (already installed) should process recently-annotated artifacts before the long tail.

## Model Provider Strategy

### Key Finding

**Gemini 2.0 Flash is cheaper than DeepSeek for this workload** — our responses are output-heavy (~600 token JSON). Gemini's output rate ($0.30/M) is 3.7× lower than DeepSeek's ($1.10/M). Per line suggestion at 800 input + 600 output tokens:

| Model | Per call | Monthly at post-launch traffic |
|---|---|---|
| Gemini 2.0 Flash | $0.00029 | ~$9 |
| Claude Haiku 4.5 | $0.003 | ~$29 |
| Claude Sonnet 4.6 | $0.012 | ~$110 |

DeepSeek excluded — data sovereignty concerns and availability issues during high-demand periods.

### Tiered Production Strategy

| Tier | Model | When | Rationale |
|---|---|---|---|
| Default | Claude Haiku 4.5 | All requests | Already integrated, no new adapter, reliable JSON, Anthropic prompt caching |
| Cost floor | Gemini 2.0 Flash | If Haiku fails eval threshold | Cheapest US-based option; one new client adapter via OpenAI-compatible endpoint |
| Quality ceiling | Claude Sonnet 4.6 | Tablets with completeness ≥ 4 AND citation_count > 10 | Reserve for high-value artifacts where quality matters most |
| Eval only | Groq + Llama 3.3 70B | Evaluation runs | Free tier covers the full eval run; sets open-weights quality floor |

Model gating (Haiku vs. Sonnet) is computed in the assembler from facts already in the bundle — no extra DB queries.

### Provider Adapter Architecture

One new file covers all non-Anthropic providers:

```python
# core/agent/openai_compat_client.py
# Covers: Groq, Together AI, Fireworks, Gemini (via their OAI-compatible endpoint),
#         future providers with /v1/chat/completions
class OpenAICompatClient:
    def __init__(self, base_url: str, api_key: str, model: str): ...
    def complete(
        self,
        system_prompt: str,
        user_message: str,
        max_tokens: int,
        json_output: bool = False,
        json_schema: dict | None = None,
    ) -> CompletionResult: ...
```

Model swaps are a one-line change in the service layer. `agent_outputs.model` tracks every generation, so quality differences are auditable by model over time.

### On-Server Inference

Current Hostinger VPS is CPU-only. Llama 3.3 70B at CPU speed is ~8 tok/s — a 600-token response takes 75 seconds, unacceptable for interactive use. Revisit when a GPU node is available. The OpenAI-compatible adapter already supports Ollama endpoints when they exist.

### Long-Term: Fine-Tuning Path

The context-heavy grounding approach means a fine-tuned smaller model (Llama 3 8B or Qwen 2.5 7B on our eval pairs) could match Sonnet quality on this narrow task at ~1/50th the cost. Training data is accumulating in `agent_outputs` + `interaction_feedback` from day one. A future export script dumps `(fact_bundle, suggestion, correction)` triples for the fine-tune. Schema doesn't need to change.

## User Refinement Loop

### Level 1 — Signal (Phase 1)

Thumbs up / down on summary block and each line suggestion. Writes `interaction_feedback (kind='explicit_rating', payload={score: 1|0})`. Tracks quality drift; flags outputs for manual review.

### Level 2 — Correction Text (Phase 2)

Scholar clicks "Suggest a correction" → inline field → `POST /api/v2/agentic/corrections`:

```json
{
  "interaction_id": "…",
  "output_id": "…",
  "original_claim": "…",
  "correction": "…",
  "scholar_id": "… (if authenticated)",
  "evidence": "…"
}
```

Server:
1. Insert `interaction_feedback (kind='correction')`.
2. Create new `annotation_runs` row with `method='user-correction'`.
3. Supersede the relevant `agent_outputs` row.
4. Next synthesis call receives the correction as a citable fact with `source_kind='user-correction'`.

The correction is a first-class annotation — same provenance infrastructure as any scholarly annotation. The correction flywheel: corrections accumulate → future fact bundles are richer → suggestions improve without retraining. The test set gets easier over time as corrections accumulate, which is the intended behavior.

### Level 3 — Accept as Translation (built, feature-flagged OFF)

"Accept" button promotes the suggestion to a real `translations` row with `source_name='agent-suggestion-accepted'`, `annotation_run_id` tracing to the suggestion's interaction. Scholar's user ID in the annotation_run.

**Gated behind `FEATURE_ACCEPT_SUGGESTION=false`.** Ships hidden. Activated only after user research with the Assyriologist community confirms incentive model and attribution norms. All training data from accepted/rejected suggestions is preserved for that research regardless.

## Evaluation Framework

### Test Set Composition

| Stratum | Count | Selection | Purpose |
|---|---|---|---|
| High-citation | 200 | Top 200 by publication citation count | Scholarly consensus; controls for model training bias (model likely has seen these) |
| Long-tail | 200 | Random sample, < 5 citations | Tests context-only performance; the real benchmark |
| Sparse metadata | 100 | Completeness ≤ 2 | Stress-tests best-guess; evaluated separately |

Note: 200 + 200 + 100 = 500 tablets × ~14 lines/tablet with ATF = ~7,000 line-suggestion calls at the ceiling. Actual untranslated lines to test against will be 3,500–7,000 depending on stratum coverage. The 7k figure is a ceiling, not a median.

### Accuracy Metrics

**Primary: Voyage cosine similarity** — embed both suggestion and reference translation using `voyage-3-large`. Threshold: cosine ≥ 0.80 = pass. Captures semantic equivalence across valid paraphrases.

**Secondary: LLM-as-judge** — for scores 0.60–0.80, ask Claude Haiku: "Does suggestion A convey the same meaning as reference B? Yes/no + one sentence." Catches valid paraphrases cosine undersells; catches plausible-sounding errors cosine oversells.

**Tertiary: Grounding compliance rate** — % of outputs passing citation validator without retry. Target: > 95% on first attempt per provider. Low compliance = model is hallucinating facts not in the bundle.

### Progressive Refinement Built Into Testing

The eval is not a one-shot gate — it is an iterative loop:

**1. Failure tagging.** Each below-threshold result is tagged by failure mode from the fact bundle: `no_lemmatization`, `mixed_language`, `sparse_genre_prior`, `unsupported_sign_form`, `competing_readings`, etc. After the first run, failure clusters drive targeted fixes — prompt first, then fact assembly.

**2. Prompt iteration loop.** Each prompt change bumps the version (`synthesis.v3`, `suggest-line.v2`). Re-run only the stratum most affected by the change. Compare new prompt_version scores against stored baselines for the same P-numbers. Regressions are visible immediately.

**3. Fact ablation.** Run the test set with individual facts removed — once without genre prior, once without similar-tablet embeddings, once without neighbor-line context. The ablation that drops the score most identifies the highest-leverage fact. Invest there before touching the prompt.

**4. Correction flywheel.** Level 2 corrections create new `annotation_runs` rows. On the next eval run, corrected tablets receive richer fact bundles → higher scores. This is real improvement, not scoring drift. Track it per stratum.

**5. Per-stratum regression tracking** via `interaction_feedback`:
```sql
SELECT prompt_version, model, AVG(payload->>'cosine') AS avg_score, COUNT(*)
FROM interaction_feedback
WHERE kind = 'eval_run'
GROUP BY prompt_version, model
ORDER BY MIN(created_at);
```

Loop order: **eval → tag failures → pick highest-frequency tag → fix (prompt or fact assembly) → bump version → re-run affected stratum → compare delta → repeat.** Each cycle is ~1 hour of work + a few dollars of API cost. After 4–5 cycles, diminishing returns signal readiness for Assyriologist involvement.

### Training Data Retention

`agent_outputs`, `interaction_feedback`, and `agent_interactions` are retained indefinitely. This is the dataset for future fine-tuning. Future export: `(fact_bundle, suggestion, correction)` triples. Schema doesn't change.

### Phase 1 — Artifact Summaries

| Severity | Issue | Fix location |
|---|---|---|
| Critical | Best-guess gate never fires for sparse tablets (no lemma_bag embedding → cosine < 0.72) | `assemble_artifact_facts`: fall back to designation embedding, lower threshold to 0.65 |
| Critical | Genre fact uses raw `artifacts.genre`, not normalized `genre_canon` | `assemble_artifact_facts`: join through `genre_canon` |

| Likely | `[n]` citations don't show source name — scholars can't verify without looking up annotation_run | Render facts as `[1] period: Ur III (CDLI catalog)` |
| Edge | First-20-signs fact is opaque to non-specialist models | Decode each sign to its most common reading from OGSL |

### Phase 2 — Line Translation Suggestions

| Severity | Issue | Fix location |
|---|---|---|
| Critical | Neighbor context is tokens on same line, not neighboring lines | `assemble_line_facts`: query ±2 lines, not ±3 token positions |
| Critical | Genre prior doesn't exist — no `entity_type='genre_prior'` in `entity_embeddings` | Build `genre_period_lemma_prior` materialized view before Phase 2 |

## Non-Negotiables

1. **No claim without a citation.** Validator rejects any output with an uncited factual claim. One retry, then graceful degraded response.
2. **Suggestions are never styled as scholarly translations.** Always carry the AI badge. Level 3 acceptance is the only path to removing it — and it's off by default.
3. **Model + prompt version tracked per row.** Both non-nullable in `agent_outputs`. Old outputs remain valid until superseded; quality differences are auditable by model.
4. **Two-tier rule holds.** Web layer → API → DB. No psycopg from `app/`. Batch CLI calls the API or core/agent directly.
5. **Competing interpretations coexist.** Unique index enforces one active row per (target, type, focus, prompt_version, model) — not one interpretation per target.
6. **Corrections are first-class annotations.** Level 2 correction = real `annotation_runs` row with `method='user-correction'`. Full provenance.
